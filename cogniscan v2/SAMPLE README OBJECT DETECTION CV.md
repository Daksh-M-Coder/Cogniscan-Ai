## 🧠 Overview — The Sim2Real Problem

**Task**: Detect a single object — **a Cheerios cereal box** — in real-world photographs.

**The twist**: We have **zero real training images**. Every training image is a photorealistic synthetic render generated by **Duality AI's FalconEditor** platform. The model must bridge the gap between synthetic perfection and real-world messiness.

```
Train on this:                  Get evaluated on this:
────────────────                ────────────────────────
Perfect lighting      →         Random room lighting
Clean textures        →         Reflections, shadows
Sharp edges           →         Motion blur, compression
Simple backgrounds    →         Cluttered real scenes
```

This is called the **Sim2Real Domain Gap** — arguably the hardest problem in modern computer vision. The model tends to learn synthetic artifacts (perfect shadows, unrealistic specularity) instead of the actual object features. Our phased training strategy is designed specifically to close this gap.

### Target Object

```
Class 0: cheerios (Cheerios brand cereal box)

Detection type: Bounding box (YOLO format)
Output format: [class_id  x_center  y_center  width  height]  (normalized 0–1)
```

### Why YOLOv8s?

| Model | Speed | Accuracy | VRAM (batch=16) | Choice |
|---|---|---|---|---|
| YOLOv8n | Fastest | Lowest | ~1.5 GB | — |
| **YOLOv8s** | **Balanced** | **Good** | **~3.5 GB** | **✅ Our choice** |
| YOLOv8m | Slower | Higher | ~5.5 GB | Phase 4+ option |
| YOLOv8l | Slow | Highest | >6 GB | VRAM limited |

YOLOv8s fits our RTX 3050 6GB with batch=16 at 640px resolution while providing a strong pretrained COCO feature backbone — ideal for fine-tuning.

## 📦 Dataset Overview

We have **three synthetic training sources** from the Duality AI FalconEditor platform, each targeting a different aspect of the Sim2Real gap:

| Dataset | Images | Purpose | What It Fixes |
|---|---|---|---|
| **Base** (`ObjectDetectionDataset`) | 1000 (862 train / 138 val) | Core detection learning | Baseline object recognition |
| **Furniture** | 300 | Background diversity | Indoor scene realism (tables, shelves) |
| **Plants** | 300 | Occlusion + texture | Visual complexity, partial hiding |
| **syntheticDataWorks** | Archive only | Duality AI reference | Not used for training |

**Combined training potential (Phase 02+):**
```
862 (base) + 300 (furniture) + 300 (plants) = 1,462 training images
```

The dataset design is intentional: each supplemental dataset attacks one specific weakness in the domain gap. Furniture teaches realistic background contexts; Plants teaches the model to handle partial occlusion and cluttered surrounding textures.

## 📊 Phase 02 — Full Dataset + Domain Augmentation

> **Goal:** Attack the Sim2Real gap on two fronts: (1) expand training to 1,462 images including furniture/plants backgrounds, (2) activate albumentations domain augmentations to simulate real camera imperfections.

### What Changed from Phase 01

| Parameter | Phase 01 | Phase 02 | Reason |
|---|:---:|:---:|---|
| **Init weights** | `yolov8s.pt` (COCO) | `phase01_best_model_iou0.9950.pt` | Fine-tune chain — not from scratch |
| **Training images** | 862 | **1,462** | +Furniture (300) + Plants (300) |
| **Epochs** | 50 | **30** | Already converged; avoid overfit |
| **LR0** | 0.0001 | **0.00005** | Fine-tuning from converged weights |
| **Mosaic** | 0.4 | **0.2** | Less spatial distortion for Sim2Real |
| **Degrees (rotation)** | 0° | **5°** | Real cameras aren't level |
| **HSV_V** | 0.4 | **0.6** | Wider brightness+contrast jitter |
| **auto_augment** | None | **`albumentations`** | Blur + noise + JPEG compression |
| **erasing** | None | **0.3** | Occlusion patch simulation |
| **mixup** | 0.0 | 0.0 | Still disabled (bad for Sim2Real) |

### Expert-Corrected Augmentation Bug

Phase 01 design had a critical silent bug — variables `NOISE=0.2`, `JPEG_QUALITY=75`, `BLUR_LIMIT=0.3` were defined but **never passed to `model.train()`** (not native YOLO params). Phase 02 fixes this with the correct approach:

```python
# WRONG (Phase 01 spec — variables defined but never applied):
BLUR_LIMIT = 0.3      # → Never reached model.train()
NOISE = 0.2           # → Never reached model.train()

# CORRECT (Phase 02 — actual YOLO/albumentations params):
auto_augment = "albumentations"   # Activates GaussianBlur + GaussNoise + ImageCompression
erasing = 0.3                     # Random rectangular occlusion patches
hsv_v = 0.6                       # Real YOLO brightness+contrast param (bumped from 0.4)
```

### Key Analysis

**Why mAP50 held at 0.9950 but mAP50-95 dropped?**

mAP50 asks: *"Is the box roughly in the right place?"* → Still 0.9950 — the model never loses the object.

mAP50-95 asks: *"Is the box pixel-perfect?"* → Dropped to 0.8955 because Gaussian blur and JPEG compression deliberately blur image edges, making tight localization harder. This is the correct trade-off: we sacrifice synthetic pixel-precision to force the model to learn object-level features that survive real-world degradation.

**Why epoch 1 dropped to 0.9747 then quickly recovered?**

Three simultaneous shocks hit the model at epoch 1: (1) 600 new images it has never seen (furniture/plants), (2) all images now degraded by blur/noise/JPEG, (3) lower LR (0.00005 vs 0.0001) slows adaptation. Recovery to 0.9950 by epoch 6 confirms the Phase 01 init was a strong starting point.

**mAP50-95 late-epoch recovery (0.835 → 0.981):**

Despite the augmentation pressure, mAP50-95 climbed from 0.835 (epoch 1) to 0.981 (epoch 30) — nearly recovering Phase 01's 0.9858. This means the model learned to be simultaneously augmentation-robust AND precisely-localizing. **This is the hallmark of better Sim2Real generalization.**

## 🔬 Technical Deep Dive

### YOLOv8s Architecture Overview

```
Input Image (640×640×3)
        ↓
Backbone: C2f (Cross Stage Partial with 2 bottlenecks)
  ├── Layer 1-4:  Feature extraction (P3: 80×80)
  ├── Layer 5-6:  Deeper features    (P4: 40×40)
  └── Layer 7-9:  Semantic features  (P5: 20×20)
        ↓
Neck: PAN-FPN (Path Aggregation Network + Feature Pyramid)
  ├── Top-down: rich semantics from P5 → P4 → P3
  └── Bottom-up: spatial precision from P3 → P4 → P5
        ↓
Head: Decoupled anchor-free detection head
  ├── (80×80 grid) — small objects (far away)
  ├── (40×40 grid) — medium objects
  └── (20×20 grid) — large objects (close up)
        ↓
Output: [class_id  cx  cy  w  h  confidence]  per detection
        ↓
NMS (Non-Maximum Suppression) → Final detections
```

### Loss Functions Used

| Loss | What It Measures | Phase 01 Final |
|---|---|---|
| **Box Loss (CIoU)** | Precision of bounding box coordinates | 0.1690 |
| **Cls Loss (BCE)** | Certainty of class label "cheerios" | 0.1336 |
| **DFL Loss** | Distribution of exact pixel edge positions | 0.7649 |

### Key Metrics Explained

| Metric | Formula | What It Means |
|---|---|---|
| **mAP50** | Area under PR curve @ IoU≥0.50 | Is the box roughly in the right place? |
| **mAP50-95** | Mean of AUC @ IoU 0.50, 0.55, ..., 0.95 | Is the box tightly aligned? (strict) |
| **Precision** | TP / (TP + FP) | Of all detections made, how many were correct? |
| **Recall** | TP / (TP + FN) | Of all actual boxes, how many were found? |
| **IoU** | Intersection / Union | Overlap between predicted and ground truth box |

### Why Mixup is DISABLED

Mixup creates ghost images by blending two training images:

```
Mixed image = α × Image_A + (1-α) × Image_B
Mixed label = α × Label_A + (1-α) × Label_B
```

For Sim2Real, this creates a problem: the blended images look **even less like real photos** than the already-synthetic renders. The model learns to identify objects in impossible, overlapping scenes that exist nowhere in reality. Result: worse generalization to real test images. This was documented in Duality AI's own notes from the reference `syntheticDataWorks` repository.

## 🔧 Infrastructure Notes

### GPU Setup (Critical)

Standard `pip install torch` on Windows downloads a **CPU-only build**. This is a silent failure — training appears to work but at 10× lower speed on CPU.

**Correct installation:**
```bash
pip install torch torchvision --index-url https://download.pytorch.org/whl/cu121
```

**Verify with:**
```bash
python SYSTEM_CHECK/gpu_verify.py
```

### Windows Multiprocessing Fix

PyTorch's DataLoader spawns worker subprocesses. On Windows, without an `if __name__ == "__main__":` guard, each worker re-imports and re-executes the entire script, causing an infinite spawn loop crash. All our training scripts contain this guard wrapping all runtime logic.

### VS Code Import Warnings

VS Code's Pylance/Pyre2 may show "cannot find import of ultralytics/torch" warnings in `.py` files. This is NOT a code error — it means VS Code is pointed at system Python instead of the venv. Fix: `Ctrl+Shift+P` → `Python: Select Interpreter` → select `venv\Scripts\python.exe`.
